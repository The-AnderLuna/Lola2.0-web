## 3. Correcciﾃｳn de Bugs Crﾃｭticos de Checkout y Lﾃｳgica (Hotfixes)
- **Bug de "El horario ya fue tomado" en reservas cruzadas:** Solucionado. Existﾃｭa una desincronizaciﾃｳn entre cﾃｳmo `disponibilidad` y `bloqueo-temporal` armaban el Tetris para reservas compartidas complejas (donde Titular y Amiga cruzaban tiempos con el mismo profesional). `disponibilidad` era flexible y permitﾃｭa que los primeros servicios "flotaran" para encontrar el encaje perfecto, mientras que `bloqueo-temporal` era rﾃｭgido con el primer servicio, causando que fallara (error HTTP 409). Se actualizﾃｳ `bloqueo-temporal` para usar exactamente la misma flexibilidad lﾃｳgica, pero validando que el grupo en conjunto inicie a la hora solicitada.
- **Bug de Checkout en Reserva Compartida (Citas en Limbo):** Identificado y resuelto un bug crﾃｭtico donde el frontend no enviaba el identificador interno (`uid`) en el payload de bloqueo. Esto causaba que la API devolviera las citas pre-bloqueadas sin `uid`, impidiendo al frontend hacer match entre los servicios del carrito y las filas de la base de datos. Como resultado, las reservas nunca pasaban a estado `PRE_AGENDADA` y se quedaban atascadas en `BLOQUEO_TEMPORAL`. Al enviar y mapear el `uid` correctamente, todo el flujo se desbloqueﾃｳ (esto tambiﾃｩn solucionﾃｳ que las horas individuales no se mostraran).
- **Bug de Cupones sin Lﾃｭmites (Usos Infinitos):** Solucionado a dos niveles. 
  1. La API de validaciﾃｳn no leﾃｭa el ID del cupﾃｳn correctamente (arreglado en el frontend). 
  2. La tabla `cupones_historial` no existﾃｭa en la base de datos, lo que causaba que el backend ignorara en silencio las inserciones de historial y permitiera a los clientes usar el mismo cupﾃｳn repetidamente. Se generﾃｳ el script SQL para crear la tabla y se fortaleciﾃｳ la API de `checkout` para que cualquier error de Supabase se registre en los logs y no vuelva a fallar en silencio.
- **Validaciﾃｳn del Sistema Anti-Fraude:** Se confirmﾃｳ en producciﾃｳn que la prevenciﾃｳn de dobles reservas funciona perfectamente (retornando un error HTTP 403) si un cliente intenta agendar mientras tiene otra cita pendiente de pago.

## 4. Estabilizaciﾃｳn de Base de Datos y APIs

## 5. Automatizaciﾃｳn y Mantenimiento Automﾃ｡tico (Cron Jobs)
Para evitar que la base de datos se llene de "basura" por intentos fallidos de reserva (bloqueos temporales abandonados y preagendas expiradas por falta de pago), se habilitaron tareas programadas automﾃ｡ticas directamente en el motor de PostgreSQL:
- **Activaciﾃｳn de `pg_cron`**: Se instalﾃｳ y habilitﾃｳ la extensiﾃｳn `pg_cron` en la base de datos de producciﾃｳn en Supabase.
- **Job de Liberaciﾃｳn Dinﾃ｡mica (`limpieza-locks-frecuente`)**: Configurado para ejecutarse **cada 10 minutos** (`*/10 * * * *`). - **Job de Purga Nocturna (`borrado-basura-nocturno`)**: Configurado para ejecutarse **todos los dﾃｭas a las 2:00 AM** (`0 2 * * *`).
